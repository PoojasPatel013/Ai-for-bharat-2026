OASIS — Self-Healing Documentation Engine

Welcome! This is the official documentation for OASIS v0.1.0.

OASIS is a GitHub/GitLab bot that treats documentation like code. It automatically validates code snippets embedded in your documentation during pull request workflows, detects bugs and breaking changes, and generates corrected versions — maintaining a Single Source of Truth for your project documentation.

Key Features:

  • Automatic Validation — Validates all code snippets in documentation on every pull request
  • Hybrid Analysis — Static analysis (AST/compile) + AI-powered reasoning via Amazon Bedrock
  • Multi-Language Support — Python, JavaScript, TypeScript, Java, Go, Rust, C/C++, and more
  • Auto-Correction — AI-powered fix suggestions posted as PR comments
  • GitHub/GitLab Integration — Seamless webhook-based bot integration
  • Secure by Design — HMAC signature verification, no code execution, scoped permissions
Powered by Amazon Bedrock — OASIS uses Claude 3 via Amazon Bedrock for intelligent code reasoning, with automatic fallback and graceful degradation when AI is unavailable.

Configuration — .doc-healing.yml

To configure OASIS for your repository, create a .doc-healing.yml file in the root of your project. This file controls which files OASIS scans, which languages are supported, and how validation behaves.

Step 1: Create the file

In your project root, create .doc-healing.yml:

.doc-healing.yml
enabled: true

documentation:
  include:
    - "docs/**/*.md"
    - "README.md"
    - "CONTRIBUTING.md"
  exclude:
    - "venv/**"
    - "node_modules/**"
    - "docs/archive/**"

languages:
  python:
    enabled: true
    timeout: 30
    dependencies:
      - "requests"
      - "pytest"
  javascript:
    enabled: true
    timeout: 15

validation:
  autoCorrect: true
  confidenceThreshold: 0.8
  blockOnFailure: true

Step 2: Understand each field

Field Type Description
enabled boolean Master switch — set to false to disable OASIS entirely
documentation.include list Glob patterns for files to scan (e.g. docs/**/*.md)
documentation.exclude list Glob patterns for files to skip
languages.*.enabled boolean Enable/disable analysis for a specific language
languages.*.timeout integer Max seconds to spend analyzing snippets in this language
languages.*.dependencies list Packages to install before validation (Python only)
validation.autoCorrect boolean Whether OASIS should suggest auto-fixes
validation.confidenceThreshold float Minimum AI confidence to suggest a fix (0.0–1.0)
validation.blockOnFailure boolean Block PR merge if broken code is detected

Step 3: Commit and push

Terminal
$ git add .doc-healing.yml
$ git commit -m "Add OASIS configuration"
$ git push origin main

OASIS will now read this file on every PR to determine which documentation files to scan.

Tip: If no .doc-healing.yml is found, OASIS defaults to scanning all .md files with Python analysis enabled.

Webhook Setup

Webhooks are how GitHub/GitLab notifies OASIS about events in your repository. When a Pull Request is opened or updated, GitHub sends an HTTP POST to OASIS, which then analyzes the PR and posts a comment with results.

What is a Webhook?

A webhook is a real-time HTTP callback. Instead of you polling GitHub for changes, GitHub pushes events to your application automatically. Think of it as a notification system: "Hey, someone just opened a PR — here are the details."

Step-by-Step: Add a GitHub Webhook

  1. Go to Repository Settings

    Navigate to your repository on GitHub → Settings → Webhooks → Add webhook

  2. Set the Payload URL

    Enter the OASIS endpoint:

    Payload URL
    http://doc-healing-alb-1100630618.ap-south-1.elb.amazonaws.com/webhooks/github

    For GitLab, use /webhooks/gitlab instead.

  3. Set Content Type

    Choose application/json from the dropdown.

  4. Set the Secret

    Enter your webhook secret token. This is used for HMAC-SHA256 signature verification to ensure requests genuinely come from GitHub.

  5. Select Events

    Under "Which events would you like to trigger this webhook?", select Let me select individual events and check:

      ✅ Pull requests
  6. Click "Add webhook"

    GitHub will send a ping event to verify the connection. You should see a green checkmark ✅.

Verification Flow

When OASIS receives a webhook, it verifies the X-Hub-Signature-256 header using HMAC-SHA256:

Signature Verification (Python)
expected = "sha256=" + hmac.new(
    webhook_secret.encode(), body, hashlib.sha256
).hexdigest()
if not hmac.compare_digest(signature, expected):
    raise HTTPException(status_code=401)

This ensures no untrusted party can trigger your bot.

Your First PR

Once you've added the .doc-healing.yml and set up the webhook, you're ready to test OASIS.

Step 1: Create a branch

Terminal
$ git checkout -b test-oasis

Step 2: Add broken code to your README

Edit README.md and add a code block with intentional bugs:

README.md
```python
def greet(name, age):
    return "Hello " + name

# Bug: missing argument 'age'
result = greet("Alice")
print(result)
```

Step 3: Push and open a PR

Terminal
$ git add README.md
$ git commit -m "Test OASIS: add broken code snippet"
$ git push origin test-oasis

Open a Pull Request from test-oasis → main.

Step 4: OASIS analyzes automatically

Within seconds, OASIS will:

  1. Receive the webhook from GitHub
  2. Fetch the PR diff and extract all code blocks from changed .md files
  3. Run static analysis on each code snippet (Python AST + compile, JS heuristics, generic bracket matching)
  4. If errors are found, optionally enhance the analysis with Amazon Bedrock AI
  5. Post a PR comment listing all issues found, with error types, line numbers, and suggested fixes

If OASIS has already commented on the PR, it updates the existing comment instead of creating a duplicate.

How the YML Configuration Works

The .doc-healing.yml file is read by OASIS on every PR event. It controls the entire behavior of the bot.

Processing Order

  1. OASIS receives a webhook event
  2. Fetches .doc-healing.yml from the PR's head branch
  3. If enabled: false, OASIS skips the PR entirely
  4. Filters changed files using documentation.include and documentation.exclude globs
  5. For each matching file, extracts code blocks tagged with languages listed under languages
  6. Applies the validation settings to determine behavior

Glob Pattern Examples

Pattern Matches
docs/**/*.md All .md files recursively under docs/
README.md Only the root README
**/*.rst All reStructuredText files in the entire repo
!docs/archive/** Exclude everything under docs/archive/

Confidence Threshold

The confidenceThreshold (0.0–1.0) controls when OASIS suggests fixes:

  • 0.75 — Static analysis fix (high confidence, reliable)
  • 0.90 — Bedrock AI fix (AI-enhanced, verified against static analysis)
  • Fixes below the threshold are not suggested to avoid false positives

How Webhooks Work

Webhooks are the backbone of OASIS integration. Here's a detailed explanation of the entire webhook lifecycle.

1. GitHub Fires the Event

When a PR is opened, GitHub constructs a JSON payload containing the PR metadata, changed files, and repository information. It then sends an HTTP POST request to the configured webhook URL.

2. OASIS Authenticates

OASIS reads the X-Hub-Signature-256 header and computes its own HMAC-SHA256 hash of the request body using the shared secret. If the signatures don't match, the request is rejected with a 401 Unauthorized response.

3. Task Enqueuing

The authenticated payload is enqueued into the task queue (Redis in production, in-memory for development). This ensures the webhook response returns immediately (< 1 second) while analysis happens asynchronously.

4. Worker Processing

A worker picks up the task and executes the full flow: fetch PR files, categorize, extract code, analyze, heal, and post the comment. This can take 5–30 seconds depending on PR size.

5. Duplicate Protection

If OASIS has already commented on the PR, it updates the existing comment instead of posting a new one. This prevents comment spam when new commits are pushed.

Duplicate detection logic
BOT_SIGNATURE = "🏜️ **OASIS"
for comment in existing_comments:
    if BOT_SIGNATURE in comment["body"]:
        # PATCH existing comment instead of POST new

Analysis Methods

OASIS uses a hybrid approach combining reliable static analysis with AI-powered reasoning.

Layer 1: Static Analysis (Always Available)

This layer uses built-in language tools and pattern matching. It runs locally with no external dependencies and is 100% deterministic.

Python Example — AST Analysis
# Detects: SyntaxError, wrong arg count, type mismatches
tree = ast.parse(code)
for node in ast.walk(tree):
    if isinstance(node, ast.Call):
        # Check argument count against function definition
        ...

Layer 2: Amazon Bedrock AI (Optional Enhancement)

When available, OASIS sends the code and static analysis results to Claude 3 via Amazon Bedrock for deeper reasoning. The AI can understand intent, suggest better fixes, and explain problems.

  • Not a dependency — OASIS works perfectly with static analysis alone
  • Graceful fallback — If Bedrock is unavailable (billing, throttling), static analysis results are used
  • Confidence scoring — AI fixes get 0.90 confidence vs 0.75 for static-only fixes

Layer 3: Diff Scanning (New!)

OASIS also scans the raw diff patch of documentation files for code-like patterns outside of fenced code blocks. This catches broken code written as plain text.

Future Vision

OASIS is actively evolving. Here is our roadmap:

🎯 Short-term (v0.2)

  • Auto-commit fixes — Instead of just commenting, OASIS will push fix commits to the PR branch
  • PR status checks — Block merge if broken code is detected (GitHub Check Runs API)
  • Richer JS/TS analysis — Using node --check and ESLint integration
  • Diff-aware healing — Only analyze and fix newly added/changed code

🚀 Mid-term (v0.3)

  • Multi-repo support — Manage OASIS across an entire GitHub organization
  • Custom rules — Define custom analysis rules per-project
  • CI/CD integration — GitHub Actions workflow for local analysis
  • Code mapping — Track relationships between code changes and documentation

🌍 Long-term Vision

  • Documentation drift detection — Detect when API changes make docs outdated
  • Auto-generation — Generate documentation from code changes
  • Community marketplace — Share and install custom analysis plugins
  • Multi-LLM support — Support for OpenAI, Gemini, and other models alongside Bedrock

Transparency

OASIS is committed to being transparent about how it works, its limitations, and its data handling practices.

What OASIS Accesses

  • PR metadata — PR number, title, changed files list
  • File contents — Only the files changed in the PR (fetched via GitHub API)
  • Diff patches — The diff of changed files to identify new/modified code

What OASIS Does NOT Do

  • ❌ Does not execute your code — all analysis is static or AI-based
  • ❌ Does not store your source code permanently
  • ❌ Does not have write access to your repository (only posts comments)
  • ❌ Does not access files outside the PR scope

AI Transparency

  • Every PR comment clearly states whether fixes came from static analysis or Bedrock AI
  • Confidence scores are included so you know how certain the fix is
  • AI suggestions are always marked with "Enhanced fix using Claude 3 via Amazon Bedrock"
  • If AI is unavailable, the comment explicitly states only static analysis was used

Open Source

OASIS aims to be fully open source. All analysis logic, webhook handling, and API code is available for review. No hidden processing or external calls beyond the documented ones (GitHub API + Bedrock API).
